## Resolved in this pass (2026-07-04 — "make the workflow safe and demo-clean")

User asked to close the remaining gaps confirmed still open by the same-day verification pass above: PDF watermark/gate, section authorization, backend validation severities, prohibited-language backend enforcement, and the remaining missing audit events. All five implemented; `npx tsc --noEmit` re-run in both repos after each change shows the same pre-existing baseline error set as before this pass — no new errors introduced (one adjacent pre-existing type bug in `submit-to-hdqa`'s audit-log call was fixed in passing since it sat directly inside a function this pass already modified).

1. **5.15 PDF workflow-state gate — now a watermark, not a hard block.** `EvalPdfData.isDraftPreview` (new) is computed in `pdf.ts` from `!FINAL_STATUSES.has(evaluation.status)` (`FINAL_STATUSES = COMPLETE | SUBMITTED | ACCEPTED`) and rendered by `NCOERTemplate.tsx` as a diagonal, low-opacity "DRAFT — NOT FOR OFFICIAL USE" overlay. Chose watermark over a hard block (matching the audit's own suggested alternative) so raters can still preview a PDF mid-workflow — the export is just unmistakably marked as non-final rather than looking identical to a signed, submitted NCOER.
2. **5.13 Section/eval authorization — the last unauthorized evaluation-mutation routes closed.** `PATCH /evaluations/:id` (top-level fields) and `PATCH /evaluations/:id/sections/:section` both now call `requireEvalChainRole(evaluationId, user, ["RATER", "SENIOR_RATER", "REVIEWER"])` — previously either route accepted a write from any authenticated user regardless of chain membership. Deliberately still allows both RATER and SENIOR_RATER (and REVIEWER) so the intentional "parallel review" behavior (SR can view/edit before the rater signs) is preserved rather than accidentally locked down while closing the access-control gap — confirmed against the two real frontend callers (`duty/page.tsx` = rater, `senior-rater/page.tsx` = SR), both fall inside the allowed role set.
3. **5.14 Validation severity taxonomy — flat WARNING/INFO replaced with a 4-tier system.** `consistency-check.ts`'s `FlagSeverity` is now `"BLOCKING_ERROR" | "CONFIRMATION_REQUIRED" | "WARNING" | "INFO"`, plus a new `hasBlockingErrors(flags)` helper so callers don't re-derive the check ad hoc. `UNSUPPORTED_CLAIM` was upgraded from `WARNING` to `CONFIRMATION_REQUIRED` (the rater must explicitly confirm/resolve an unsupported factual claim, not just see it in passing). `submit-to-hdqa` now re-runs the full consistency check and rejects with 409 if any `BLOCKING_ERROR` flag remains — closing the loop for bullets saved before enforcement existed. Frontend `ConsistencyCheckModal.tsx` and the `ConsistencyFlag` type were updated in the same pass to recognize the new severities (see item 4 below — this was a real regression risk caught mid-pass, not an afterthought).
4. **5.14 Prohibited-language backend enforcement — the dead frontend checker is now the live backend gate.** New `src/lib/ai/prohibited-language.ts` (backend) is a byte-for-byte port of `ees2-frontend/src/lib/ai/prohibited-language.ts`'s pure logic (first-person, protected-class terms, SSN, marital status, political references, future-tense promises, over-length) — that module's own header comment claimed it "runs in three places" but was previously imported nowhere (MVP audit 5.14 finding). `PATCH /evaluations/:id/sections/:section` now runs `checkBulletQuality()` on every `finalBullets` entry and rejects the write with 400 if any bullet contains an ERROR-severity issue — a direct API call can no longer bypass this the way it could when the check only existed client-side. `PROHIBITED_LANGUAGE` flags are also surfaced (again, non-authoritatively) in the pre-submission consistency check, resolving the previously-documented dead-UI-branch contradiction: `ConsistencyCheckModal.tsx` already had a label for this exact flag code that the backend had never emitted until now.
5. **5.16 Missing audit events — remaining gaps closed.** Added `auditLog.create` calls for: artifact upload (`ARTIFACT_UPLOADED`), artifact flag (`ARTIFACT_FLAGGED`), artifact delete (`ARTIFACT_DELETED`), support-form entry creation (`ENTRY_CREATED`), and PDF export (`PDF_EXPORTED`, includes `isDraftPreview` in metadata). Real audit call-site count is now 13 (up from the 9 confirmed in the same-day verification pass), and the only capability-level actions still unaudited are ones not in scope for this pass (see "Not yet addressed" below).

Schema: no migration needed this pass — all five fixes are route/lib/component-level, no new persisted fields.

## Verification pass (2026-07-04) — spot-checking the third-pass claims

Eight targeted questions, each answered by re-reading the actual route/component/schema code (not by trusting the prior pass's own narrative). Results below; the audit table and cross-cutting notes further down have been updated to match.

1. **Can a final accepted bullet open a View Sources panel showing originating suggestion, entries, and artifacts? YES.** `BulletCard.tsx`'s "View source" toggle renders `provenance.sourceSnapshot` (source entry `rawText` + artifact captions), resolved entirely client-side from the embedded snapshot — zero extra API calls, immune to later entry edits/deletes. The full chain (`suggestionId` → `sourceEntryIds` → `sourceSnapshot`) is present in the underlying `EvalSection.bulletProvenance` data (`BulletProvenanceEntry` type), though the UI itself only surfaces the entry text/captions, not the raw suggestion ID — sufficient for the rater-facing use case.
2. **Does accepting the same suggestion twice produce only one final bullet? YES.** `PATCH /support-form-uploads/bullets/:bulletId` wraps the status flip and the section's `finalBullets` append in a single `Serializable` Prisma transaction, gated by a conditional `updateMany({ where: { id, status: "PENDING_REVIEW" } })`. A second concurrent/duplicate call matches 0 rows and throws a 409 inside the transaction (rolled back) — genuinely idempotent, not just idempotent-by-convention.
3. **Does an edited/deleted source entry leave the original generation snapshot intact? YES, by construction — with a caveat.** `sourceSnapshot` is a plain JSON value copy (`{ entryId, rawText, artifactCaptions }`) written into `AIBulletSuggestion` at generation time in both `generate-from-entries` and `generate-scratch` — it holds no foreign key/relation back to `SupportFormEntry`, so it cannot be affected by later mutation of the source row. Caveat: there is currently **no route to edit or delete a `SupportFormEntry`'s `rawText` at all** (only `confirmationStatus` can change via the confirm route) — so this guarantee hasn't been exercised against a real mutation yet, but the design would hold regardless once such a route exists.
4. **Does unsupported-fact detection catch a fabricated number? YES.** `checkUnsupportedFacts()` in `unsupported-fact-check.ts` extracts 2+ digit numbers via a dedicated pattern, normalizes all source facts, and flags any number whose digit-only form isn't present in the normalized source text — directly matches the previously-verified live "12 exercises/95%" test.
5. **Is the performance timeline actually imported and reachable from the UI? YES.** `PerformanceTimeline.tsx` is imported by `app/(dashboard)/evaluations/[id]/timeline/page.tsx`, and `SectionNav.tsx`'s `STEPS` array includes a `"timeline"` entry (between Duty Description and Character) — genuinely wired into the sidebar nav, not a dead component.
6. **Are artifact upload routes rating-chain authorized and file-type/size protected? PARTIAL.** File-type/size protection: yes — multer memory storage, 20MB limit, `fileFilter` restricted to `application/pdf`/`image/jpeg`/`image/png`/`image/webp`, applied consistently on both the support-form-entry artifact route and the whole-document upload route. Authorization: artifact routes use `requireSupportFormEntryOwner`/`requireArtifactOwner` — this is **soldier self-ownership** (the soldier who owns the support form), **not rating-chain-role authorization** (rater/SR/reviewer). That's the intentionally correct model for a soldier uploading their own evidence, but it means the phrase "rating-chain authorized" doesn't literally apply here the way it does for generation/signing/PDF/milestones — worth keeping distinct in documentation to avoid implying rater/SR routes gate this.
7. **Does PDF export block or watermark incomplete evaluations? NO — confirmed still open.** `pdf.ts` only calls `requireEvalChainRole` (the chain-membership fix from a prior pass); there is no evaluation-status precondition and no watermark logic anywhere in `lib/pdf/generator.ts` or `NCOERTemplate.tsx` (grep for `watermark|DRAFT|status` in `lib/pdf/` returns nothing). A `DRAFT` evaluation with zero bullets can still successfully generate and download a mostly-empty PDF. This is exactly the 5.15 gap noted previously — unchanged.
8. **Are suggestion decisions, status transitions, artifact actions, and PDF exports audited? PARTIAL — broader than previously documented, but with real gaps.** Confirmed 9 real `auditLog.create` call sites (the earlier "only 3 call sites" note from the prior pass is now stale): `SUGGESTION_ACCEPTED`/`SUGGESTION_EDITED`/`SUGGESTION_REJECTED` (support-form-uploads.ts), `EVALUATION_STATUS_CHANGED` (status.ts), `SIGNATURE_APPLIED`/`SIGNATURE_DECLINED` + `SUBMITTED_TO_HDQA` (evaluations.ts), `MILESTONE_COMPLETED`/`MILESTONE_WAIVED` (milestones.ts), `ENTRY_CONFIRMATION_CHANGED` (support-forms.ts), plus one in delegates.ts. Still **not** audited: artifact upload (`POST .../entries/:entryId/artifacts`), artifact flag (`PATCH .../artifacts/:artifactId/flag`), artifact delete (`DELETE .../artifacts/:artifactId`), PDF export (`GET /pdf/evaluations/:id`), and support-form-entry creation.

## Resolved in this pass (2026-07-03, third pass — "harden the evidence-to-bullet lifecycle")

User asked to complete and harden: immutable source snapshots, transactional suggestion acceptance, artifact-linked provenance, unsupported-fact warnings, upload authorization verification, and the performance timeline. All six implemented; verified live where the environment allowed (Claude generation itself can't be exercised without `ANTHROPIC_API_KEY`, but every deterministic/authorization piece was).

1. **5.1 Immutable source snapshots (was a gap → now fixed).** New `AIBulletSuggestion.sourceSnapshot` (Json) captures each contributing entry's `rawText` + artifact captions *at generation time*, for both `generate-from-entries` and `generate-scratch` (the rater's own description doubles as the "source" there). Later edits/deletes of the underlying entry can no longer rewrite what a suggestion was actually generated from.
2. **5.10 Unsupported-fact warnings (was 100% MISSING → now implemented).** New `src/lib/ai/unsupported-fact-check.ts` — deterministic, regex-based extractor for numbers, percentages, dates, named schools/courses, and awards/rankings; flags any claim in bullet text not present in the source snapshot. Stored per-suggestion as `unsupportedClaims` (shown in `AIBulletPanel` before acceptance) and re-checked at pre-submission time as a new `UNSUPPORTED_CLAIM` consistency-check flag (reads the final bullet's `bulletProvenance.sourceSnapshot`, so a claim introduced by a later manual edit is still caught). **Verified live** via direct unit execution: a fabricated "12 exercises / 95%" against real source facts of "3 exercises" correctly flagged both; the real, present "572" ACFT score was correctly *not* flagged; a fabricated date + school name were both correctly flagged.
3. **5.8 Transactional, idempotent suggestion acceptance (was BROKEN → now fixed).** `PATCH /support-form-uploads/bullets/:bulletId` rewritten: a conditional `updateMany` (`where: { id, status: "PENDING_REVIEW" }`) makes a duplicate/double-click request a clean 409 instead of silently creating a second final bullet; for ACCEPTED/EDITED, the suggestion-status flip and the section's `finalBullets`/`bulletSources`/`bulletProvenance` append now happen inside one `Serializable` Prisma transaction. The route is now also chain-role-authorized (previously anyone authenticated could resolve any suggestion) and audits `SUGGESTION_ACCEPTED`/`SUGGESTION_EDITED`/`SUGGESTION_REJECTED`. Frontend (`AIBulletPanel.tsx`) updated to consume the new `{ suggestion, section }` response directly via a new `onSectionUpdated` callback, replacing the old two-call (PATCH bullet, then separately PATCH section) race-prone pattern entirely.
4. **5.9 Artifact-linked provenance (was MISSING → now implemented).** New `EvalSection.bulletProvenance` (Json) maps each final-bullet index to `{ suggestionId, sourceEntryIds, sourceSnapshot }`, written atomically by the transactional accept above. `BulletCard.tsx` now has a "View source" toggle that resolves this straight from the embedded snapshot (no extra API calls, and immune to later entry edits/deletes) to show the original entry text + artifact captions a bullet came from.
5. **5.2 Upload authorization verification (was unverified/gap → now hardened).** New `requireSupportFormEntryOwner`/`requireArtifactOwner` in `chain-auth.ts`. Applied to artifact upload (also now verifies the `:entryId` actually belongs to the `:formId` in the URL, previously unchecked), flag, and delete routes — all three previously had **zero** authorization beyond "is logged in." **Verified live**: a non-owner's upload attempt now returns 403 before ever reaching storage; the actual owner's request passes authorization (fails later only on an unrelated, pre-existing missing-Supabase-credential environment gap in this dev sandbox). Also fixed a real adjacent bug found while testing: `flaggedByServiceMember` used `z.coerce.boolean()` against multipart form-data, where JS's `Boolean("false")` is `true` — any explicit "false" string was silently treated as true. Replaced with a proper string-literal transform.
6. **5.5 Performance timeline (was dead/unwired → now implemented).** New `PerformanceTimeline.tsx` composes support-form entries + counseling sessions + milestones (reusing the existing `SuspenseTimeline` chip-strip for the milestone portion, per the audit's own "don't build a third component" note) into one filterable, chronological, read-only feed — filters for dimension, entry type, counseling visibility, and evidence presence. New `/evaluations/[id]/timeline` page, wired into both `SectionNav.tsx` (sidebar) and `SECTION_ORDER` (prev/next flow, placed right after Duty Description). The old dead `EntryTimeline.tsx`/stub was superseded rather than finished in place, since it was an unfinished, unwired stub with no composition logic.

Schema: one migration this pass, `20260703181450_evidence_lifecycle_provenance_snapshots` (added `sourceSnapshot`/`unsupportedClaims` to `AIBulletSuggestion`, `bulletProvenance` to `EvalSection`).

Baseline preserved: `npx tsc --noEmit` in both repos shows the exact same pre-existing error set as before this pass — no new errors introduced (verified twice, including after fixing two rounds of Prisma `Json`-typing issues surfaced by the new fields).

**Not yet addressed** (still open as of 2026-07-04): 5.11 milestone due/overdue list UI (auth was fixed in a prior pass; a dedicated non-timeline view still doesn't exist, though the timeline page shows the milestone chip strip), 5.14 stale-signature check + authentication-prerequisite check in `consistency-check.ts` (severity taxonomy and prohibited-language enforcement were closed in the 2026-07-04 safety pass above — these two specific checks were not). **5.13, 5.15, and 5.16 were closed in the 2026-07-04 safety pass** (section/eval authorization, PDF watermark, and the remaining audit-event gaps, respectively).

## Cross-cutting findings (apply across multiple capabilities above)

### Authorization — the single biggest systemic gap
Nearly every route inspected uses only `requireAuth` (proves *a* user is logged in) with **no verification that the user is actually a member of the relevant `RatingChain`** for the resource being touched. Confirmed concretely on: `generate-from-entries` (any entryId from any soldier), `GET /api/pdf/evaluations/:id` (any evaluation), `PATCH /milestones/:id/complete` and `/waive` (any milestone). This is not a per-capability bug — it's one missing piece of middleware/helper (e.g. a `requireChainMembership(evaluationId, allowedRoles)` guard) that should be added once and applied broadly, rather than patched ad hoc per route. Recommend doing this as its own early step since it touches 5.6, 5.11, 5.13, 5.15, and the negative-test requirements of nearly every other capability.

### Duplicate AI pipeline (pre-existing, not created by prior sessions)
`src/routes/ai.ts` + `src/lib/ai/openai.ts` + `AiGeneration` model form a complete, unreachable, parallel bullet-generation system alongside the real one (`support-form-pipeline.ts` + `claude.ts` + `AIBulletSuggestion`). Needs an explicit decision (see 5.7) before any new generation work — extending the wrong pipeline would waste effort and violate "do not create a second AI-generation pipeline."

### Two dead frontend modules, fully built but never wired
- `ees2-frontend/src/lib/ai/prohibited-language.ts` — complete prohibited-language checker, zero imports.
- `ees2-frontend/src/components/support-form/EntryTimeline.tsx` + `.../milestones/SuspenseTimeline.tsx` — timeline UI, zero imports (and `EntryTimeline` is also an unfinished stub).

These are exactly the "documented but absent" / "dead/unreachable code" categories the assignment specifically asked us to distinguish from "complete."
